# Redis

键值对（Key-Value）数据库。单线程，数据`存储在内存`中，读写速度极快，支持丰富数据结构（如List,Set）和数据过期时间。

`适合极致读写速度、临时状态存储、或者高并发计数的场景`。


## 安装启动

以 Linux CentOS 7.x 为例（Redis官方**不支持windows系统**）
1. 安装EPEL仓库和Redis依赖
```shell
sudo yum install -y epel-release
sudo yum install -y redis
```
2. 启动与设置开机自启（默认端口`6379`）
```shell
# 启动 Redis 服务
sudo systemctl start redis

# 设置开机自动启动
sudo systemctl enable redis

# 查看运行状态（确认是否成功启动）
sudo systemctl status redis
```
3. 连接 Redis
```shell
# 输入redis-cli即可进入交互界面
redis-cli

# 输入 ping，如果返回 PONG，说明连接成功。
127.0.0.1:6379> ping
PONG
```
4. 设置密码和开放远程连接
:::code-group
```bash [1.打开配置文件]
# 打开redis配置文件：redis.conf 文件
# 通常它位于 /etc/redis/redis.conf 或 /etc/redis.conf。

sudo vim /etc/redis/redis.conf
```
```bash [2.修改密码]
# 在p配置文件中搜索 requirepass（在 vim 中可以按 / 然后输入 requirepass 搜索）。
# 默认情况下，这一行是被注释掉的（前面有 # 号）
# 你需要取消注释（删掉 # 号），并将后面的值改成你自己的强密码

requirepass [自定义密码]
```
```bash [3.允许远程连接]
# 在 redis.conf 文件中搜索 bind。
# 找到类似 bind 127.0.0.1 的配置，将其修改为：
bind 0.0.0.0

# 继续搜索 protected-mode。
# 将它的值从 yes 改为 no：

```
```bash [4.重启 Redis并测试连接]
sudo systemctl restart redis

redis-cli -h <服务器IP地址> -p 6379 -a <你设置的密码>
```
:::

## 单线程架构

Redis 的单线程指的是：**它所有的核心命令（如 GET、SET、INCR 等）都是由一个主线程串行执行的**。 就像银行只有一个服务窗口，所有客户（客户端请求）都必须排队，一个一个地接受服务。

**为什么Redis单线程还能这么快？**

- `纯内存操作`：Redis 的所有数据都存储在内存中，而内存的读写速度是磁盘的成千上万倍
- `I/O 多路复用技术`：Redis使用epoll作为I/O多路复用技术的实现，再加上Redis 自身的事件处理模型将epoll中的连接、读写、关闭都转换为事件。当某个连接有数据可读或可写时，Redis 才会去处理它。不在网络I/O上浪费过多的时间，
- `避免了多线程的复杂性`：避免了多线程编程中常见的锁竞争、上下文切换和死锁以及线程切换的产生的开销。

> 单线程最大好处：由于所有命令都是在一个线程里排队执行的，这就保证了命令的原子性从而避免了数据不一致。

## 配置文件

```text
# ================= 1. 网络与基础配置 =================
# 绑定地址：生产环境建议绑定内网IP，或者配合防火墙使用 0.0.0.0(允许所有 IP 访问)
bind 0.0.0.0
# 监听端口
port 6379
# 后台守护进程运行
daemonize yes
# 最大客户端连接数
maxclients 10000
# 客户端空闲超时时间（秒），0表示不限制，建议设置以释放空闲连接
timeout 300
# TCP 保活探测间隔，官方推荐 60 秒，用于检测死连接
tcp-keepalive 60
# TCP 全连接队列大小，高并发下建议调大
tcp-backlog 65535

# ================= 2. 安全与权限配置 =================
# 设置访问密码（强烈建议设置强密码）
requirepass YourStrongPassword123!
# 禁用高危命令，防止误操作或恶意破坏
rename-command FLUSHALL ""
rename-command FLUSHDB ""
rename-command CONFIG ""

# ================= 3. 内存管理配置 =================
# 设置最大内存限制（根据服务器实际内存调整，例如分配 4GB）
maxmemory 4gb
# 内存淘汰策略：当内存达到上限时，淘汰最久未使用的键（适合纯缓存场景）
maxmemory-policy allkeys-lru

# ================= 4. 日志与持久化配置 =================
# 日志级别（debug, verbose, notice, warning），生产环境推荐 notice
loglevel notice
# 日志文件路径
logfile "/var/log/redis/redis.log"
# 数据文件存储目录
dir /var/lib/redis

# --- RDB 快照持久化（性能优先） ---
# 900秒内至少1个键改变则触发快照
save 900 1
# 300秒内至少10个键改变则触发快照
save 300 10
# 60秒内至少10000个键改变则触发快照
save 60 10000
# RDB 文件名
dbfilename dump.rdb
# 开启 RDB 压缩
rdbcompression yes

# --- AOF 日志持久化（数据安全优先） ---
# 开启 AOF 持久化
appendonly yes
# AOF 文件名
appendfilename "appendonly.aof"
# 同步频率：每秒同步一次（性能与安全的最佳平衡点）
appendfsync everysec

# ================= 5. 性能优化配置 (Redis 6.0+) =================
# 如果你的 Redis 版本是 6.0 及以上，且服务器 CPU 是多核的，强烈建议开启多线程 IO
# 开启多线程读操作
io-threads-do-reads yes
# IO 线程数（通常设置为服务器 CPU 核心数的 3/4，例如 4 核机器设置为 3）
io-threads 4
```

:::tip 配置文件默认路径


- 如果你是通过 Ubuntu/Debian 的 apt 或 CentOS/RHEL 的 yum 安装的，配置文件通常位于：
/etc/redis/redis.conf （最常见，如 Ubuntu/Debian）
/etc/redis.conf （常见于 CentOS/RHEL）
- 如果你是下载源码包手动编译安装的，配置文件通常位于：
/usr/local/etc/redis.conf
或者在你解压的 Redis 源码根目录下（如 /usr/local/src/redis-x.x.x/redis.conf）
- 如果是使用官方 Docker 镜像，默认路径通常是 /usr/local/etc/redis/redis.conf。

## 数据类型&常用命令

Redis的数据都是通过key:value形式进行存储的。

其中key是**字符串**，value可以是：
- **String**：字符串。可以存普通的文本字符串，还能存数字、甚至二进制数据（如图片、序列化后的对象）。适用于`缓存、计数器（如文章阅读量）,session共享，验证码`等场景。
- **List**：简单的字符串列表，按照插入顺序排序。适用于`消息队列、朋友圈时间线、最新文章列表`。
- **Set**：无序的字符串集合，不允许重复。适合`抽奖系统（利用随机弹出）、共同好友/共同关注（利用交集运算）、去重`。
- **SortedSet**：和 Set 类似，不允许重复，但每个元素都关联了一个分数（score），Redis 会根据分数自动进行排序。适合`排行榜（如游戏积分排行），视频播放量排行`等场景
- **Hash**：它是一个键值对（field-value）的集合，非常适合用来存储对象。适合`存储用户信息、商品详情`等对象数据

:::code-group
```bash [String]
# SET / GET：设置和获取最基础的键值对。
# 缓存一个用户的简单状态
# 这里的key为user:1001:status(下面同理)，通过冒号符分隔进行逻辑分层，方便管理和后续查询。
127.0.0.1:6379> SET user:1001:status "active"
OK
127.0.0.1:6379> GET user:1001:status
"active"


# SETEX：设置值的同时指定过期时间（非常常用，防止缓存数据永久占用内存）。
# 设置一个验证码，60秒后自动过期
127.0.0.1:6379> SETEX sms:code:1380000 60 "1234"
OK


# INCR / INCRBY：原子递增操作。因为是单线程执行，所以非常适合做高并发下的计数器。
# 商品ID为8848的浏览量每次访问加1
127.0.0.1:6379> INCR product:8848:views
(integer) 1
127.0.0.1:6379> INCR product:8848:views
(integer) 2
# 如果一次性增加了5个浏览量
127.0.0.1:6379> INCRBY product:8848:views 5
(integer) 7
```

```bash [Hash]
# HSET / HGET：设置和获取哈希表中某个字段的值。
# 设置用户1001的年龄为25岁
127.0.0.1:6379> HSET user:1001:info age 25
(integer) 1
# 获取年龄
127.0.0.1:6379> HGET user:1001:info age
"25"


# HMSET / HMGET：批量设置和获取多个字段（减少网络往返次数）。
# 一次性设置姓名和余额
127.0.0.1:6379> HMSET user:1001:info name "Alice" balance 1000
OK
# 一次性获取姓名和余额
127.0.0.1:6379> HMGET user:1001:info name balance
1) "Alice"
2) "1000"

# HGETALL：获取该对象的所有字段和值。
127.0.0.1:6379> HGETALL user:1001:info
1) "age"
2) "25"
3) "name"
4) "Alice"
5) "balance"
6) "1000"
```
```bash [List]
# LPUSH / RPUSH：从列表的左边（头部）或右边（尾部）插入元素。
# 模拟用户发朋友圈，新消息插入到队列头部
127.0.0.1:6379> LPUSH timeline:user:1001 "今天天气真好"
(integer) 1
127.0.0.1:6379> LPUSH timeline:user:1001 "中午吃了火锅"
(integer) 2


# RANGE：获取指定范围内的元素（常用于分页查询，0 代表第一个，-1 代表最后一个）。
# 获取最新的2条动态
127.0.0.1:6379> LRANGE timeline:user:1001 0 1
1) "中午吃了火锅"
2) "今天天气真好"


# LPOP / RPOP：从头部或尾部弹出一个元素（常用于消费队列中的任务）。
# 消费者从队列尾部取出最早的一条消息处理
127.0.0.1:6379> RPOP timeline:user:1001
"今天天气真好"
```
```bash [Set]
# SADD / SMEMBERS：向集合中添加元素，并查看所有元素。
# 用户1001、1002、1001（重复）给文章8848点赞
127.0.0.1:6379> SADD article:8848:likes "user:1001" "user:1002" "user:1001"
(integer) 2  # 返回2表示成功添加了2个不重复的元素
127.0.0.1:6379> SMEMBERS article:8848:likes
1) "user:1001"
2) "user:1002"


# SISMEMBER：判断某个元素是否在集合中（常用于判断用户是否点赞）。
127.0.0.1:6379> SISMEMBER article:8848:likes "user:1001"
(integer) 1  # 1表示存在，0表示不存在


# SINTER / SUNION：求交集和并集。
# 假设 user:1001 关注了 A, B, C
127.0.0.1:6379> SADD user:1001:follows "userA" "userB" "userC"
# user:1002 关注了 B, C, D
127.0.0.1:6379> SADD user:1002:follows "userB" "userC" "userD"

# 求两人的共同关注（交集）
127.0.0.1:6379> SINTER user:1001:follows user:1002:follows
1) "userB"
2) "userC"



```

```bash [SortedSet]
# ZADD：添加元素并指定分数。
# 玩家Alice得分为100，Bob得分为200
127.0.0.1:6379> ZADD game:leaderboard 100 "Alice" 200 "Bob"
(integer) 2


# ZINCRBY：增加某个元素的分数。
# Alice又赢了，分数增加50
127.0.0.1:6379> ZINCRBY game:leaderboard 50 "Alice"
"150"


# ZRANGE / ZREVRANGE：按分数从低到高（ZRANGE）或从高到低（ZREVRANGE）获取排名。
# 获取排行榜前两名（0到1），并带上分数（WITHSCORES）
127.0.0.1:6379> ZREVRANGE game:leaderboard 0 1 WITHSCORES
1) "Bob"
2) "200"
3) "Alice"
4) "150"


# ZREVRANK：获取某个元素的排名（降序，0代表第一名）。
127.0.0.1:6379> ZREVRANK game:leaderboard "Alice"
(integer) 1  # Alice排第2名（索引为1）
```
:::

键通用命令：

- `KEYS [pattern]`：查找所有符合给定模式的 Key。例如`KEYS user:*`来查找所有以`user:`开头的Key
- `EXISTS [键名]`：判断一个键是否存在。
- `DEL [键名] [键名]...`：删除一个或多个键。
- `TYPE [键名]`：查看键所代表的数据结构类型。
- `EXPIRE [键名] [秒数]`：为键设置过期时间。
- `TTL [键名]`：查看键的剩余过期时间。
    ```bash
    # 如果返回 -1，表示该 Key 存在，但没有设置过期时间（永久有效）。
    # 如果返回 -2，表示该 Key 不存在。
    127.0.0.1:6379> TTL user:1001:status
    (integer) 58
    ```
数据库管理命令：
- `SELECT [数据库索引]`：选择一个数据库。 Redis默认提供了 16 个逻辑数据库（编号 0-15）
- `FLUSHDB`：清空当前数据库的所有数据（**生产环境慎用！**）。
- `FLUSHALL`：清空所有数据库的所有数据（**生产环境慎用！**）。

## 日志

### 运行日志

主要记录服务器的运行状态、启动/关闭事件、客户端连接断开、持久化操作以及报错信息等。它不记录具体的业务命令，而是**记录 Redis 进程本身的生命周期事件**。

```bash
# 获取运行日志文件路径
redis-cli CONFIG GET logfile

# 实时监控最新的日志输出（排查故障时最常用）
tail -f /var/log/redis/redis-server.log

# 分页查看日志，支持上下滚动和按 "/" 搜索关键字
less /var/log/redis/redis-server.log
```

### 持久化日志

如果开启了 AOF 持久化（appendonly yes），Redis 会把每一次的写操作命令（如 SET、DEL、INCR 等）以 Redis 协议的格式追加到 appendonly.aof 文件中。

核心作用：
- **数据恢复**：Redis 重启时，会通过重新执行 AOF 文件中的命令来恢复数据。
- **审计与追溯**：你可以直接查看这个文件，了解历史上对 Redis 做了哪些写操作。

```bash
# 查看 AOF 文件的前 20 行
cat /var/lib/redis/appendonly.aof | head -n 20
```

### 慢查询日志

慢查询日志主要作用是记录那些执行时间超过了预设阈值的命令，帮助开发者快速定位拖慢系统的瓶颈。

慢查询的行为由配置文件(redis.conf)中两个核心参数控制：
```text{4,9}
# 设置慢查询的时间阈值，单位是微秒。默认值是 10000（即 10 毫秒）
# 如果设置为 0，会记录所有命令；如果设置为负数，则关闭慢查询日志
#  生产建议：在高并发场景下，建议设置为 1000（1毫秒）甚至更低，以便尽早发现潜在的性能隐患。
slowlog-log-slower-than 10000


# 设置慢查询日志的最大长度（条数）。默认值是 128。当达到上限时，最早的日志会被自动移除。
#  生产建议：建议调大到 512 或 1000 以上，防止慢查询过多时历史日志被过快覆盖。
slowlog-max-len 128
```

查看日志：
```bash
# 查看最近 10 条慢查询日志
127.0.0.1:6379> SLOWLOG GET 10

# 获取当前慢查询日志的总条数
127.0.0.1:6379> SLOWLOG LEN

# 清空当前的慢查询日志
127.0.0.1:6379> SLOWLOG RESET
```

:::warning 常见慢查询原因
- `KEYS *`：遍历所有键，时间复杂度 O(N)，在生产环境中是绝对禁止的。
- `SORT、SUNION、ZUNIONSTORE`：对大量数据进行排序或聚合运算，会消耗大量 CPU。
- `HGETALL、SMEMBERS`：一次性获取大集合的所有元素，当 N 的值非常大时也会变慢。
- `BigKey（大键）操作`： 如果一个 Key 存储了海量数据（例如一个包含百万级元素的 List 或 Hash），即使是简单的 DEL、SET 或 HSET 操作，也会因为需要处理大量内存而阻塞主线程。
:::
## 缓存

Redis 作为最主流的缓存中间件，其核心作用就是把频繁访问的数据存放在内存中，避免每次请求都去查询数据库，从而极大地提升系统性能并减轻数据库压力。

### 基本机制

- 查询：当应用程序需要读取数据时，首先向 Redis 发起查询。
- 命中（Hit）：如果 Redis 中存在该数据，则直接返回给应用程序，流程结束。
- 未命中（Miss）：如果 Redis 中没有该数据，则转向数据库查询。查到数据后，将数据写入 Redis（通常会设置一个过期时间），然后再返回给应用程序。

### 更新策略

当数据库中的**数据发生变更时，如何保证 Redis 缓存与数据库的数据一致性**，是缓存设计的核心。

常见的更新策略有以下三种：

| 策略类型 | 核心逻辑 | 优点 | 缺点 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- |
| 超时剔除 | 为缓存数据设置 TTL（过期时间），到期后自动删除，下次查询时再更新。 | 维护成本极低 | 存在过期时间窗口内的数据不一致 | 对一致性要求不高的场景 |
| 内存淘汰 | 依赖 Redis 的内存淘汰机制（如 LRU），内存不足时自动淘汰旧数据。 | 无需手动维护 | 一致性最差，数据何时被淘汰不可控 | 低一致性需求的场景（如店铺类型查询） |
| 主动更新 | 业务代码在修改数据库的同时，主动去更新或删除缓存。 | 数据一致性好 | 维护成本高，需处理并发下的脏数据风险 | 对一致性要求较高的核心业务 |

:::tip 最佳实践
- 查询数据时：先查缓存，未命中则查数据库，将结果写入缓存并设置 TTL。
- 写入数据时：`先更新数据库，然后再删除缓存`

**Q1：写入数据时为什么是删除缓存而不是更新缓存？**

A1：如果是更新缓存，则并发写操作场景（无查询）下可能大多更新缓存都是无意义操作，删除缓存则直接等待下次查询时再更新缓存。

**Q2：写入数据时为什么是先更新数据库再删除缓存？不能先删除缓存再更新数据库吗？**

A2：如果先删除缓存再更新数据库，在并发读写时，读请求可能会在数据库更新前把旧值重新加载到缓存中，导致数据不一致。


:::


### 缓存穿透

缓存穿透是指客户端请求`查询的数据在缓存（Redis）和数据库中都不存在`。

正常情况下，缓存未命中会去查数据库。但如果黑客恶意发起大量请求，查询根本不存在的 ID（如负数、超长乱码），这些请求会直接“穿透”缓存层，全部打到数据库上。如果并发量极大，很容易将数据库压垮。

解决方案：

#### 方案一：缓存空值（Null / 默认值）
- 原理：当数据库查询结果为空时，依然在 Redis 中**缓存一个空值（例如 "" 或特定的缺省值），并设置一个较短的过期时间**（如 2 分钟）。
- 效果：下次再有相同的非法请求时，会直接命中缓存中的空值并返回，从而保护了数据库。
- 缺点：如果攻击者每次使用随机的、不存在的 ID 进行攻击，会导致 Redis 中`产生大量无用的 Key`，消耗额外内存。 

#### 方案二：布隆过滤器（Bloom Filter）拦截
- 原理：在请求到达 Redis 之前，加一层布隆过滤器。布隆过滤器是一种概率型数据结构，它会提前把数据库中所有合法的 Key 映射进去。
- 效果：当请求到来时，先经过布隆过滤器判断。如果布隆过滤器说“不存在”，那么这个数据一定不存在，直接拦截请求；如果布隆过滤器说“存在”，再去查 Redis 和数据库。
- 优点：极大地节省了内存，且拦截效率极高。
- 缺点：`有一定的误判率`（即布隆过滤器说存在，但实际上可能不存在，不过这种情况依然会走正常的缓存未命中流程，不会造成系统崩溃），且实现相对复杂，数据删除和更新维护成本较高。

### 缓存击穿

缓存击穿通常发生在某一个非常热门的 Key（比如微博热搜第一的词条、秒杀活动的商品详情）。

这个`热点 Key 在某一时刻刚好过期失效`了，此时恰好有海量的并发请求同时来访问这个 Key。因为缓存中没有数据，这些请求会瞬间全部穿透到数据库，导致数据库压力骤增，甚至被瞬间压垮。

解决思路：**控制并发，保证只有一个线程去查数据库，其他线程等待。**

#### 方案一：互斥锁（分布式锁）
- 原理：当发现缓存未命中时，不直接去查数据库，而是先去 Redis 中尝试获取一把“分布式锁”（比如使用 SETNX 命令）。
- 效果：只有获取到锁的那个线程才有资格去查询数据库并重建缓存。其他没抢到锁的线程，只能短暂休眠后重试，或者等待锁释放。
- 优点：保证了数据的一致性，且不会浪费内存。
- 缺点：实现相对复杂，线程需要等待，还需要`处理死锁问题`（必须设置锁的超时时间）。
#### 方案二：逻辑过期
- 原理：热点数据在 Redis 中不设置物理过期时间（永不过期），但在 Value 中额外存储一个“逻辑过期时间”。
- 效果：当线程读取数据时，判断逻辑时间是否过期。如果没过期，直接返回；如果过期了，当前线程先返回旧数据，然后开启一个异步线程去后台更新缓存。
- 优点：性能极高，用户永远不需要等待。
- 缺点：会`短暂返回旧数据（存在数据不一致的窗口期）`，且需要额外的内存来存储逻辑时间。

### 缓存雪崩

缓存雪崩是指`大量（甚至全部）的缓存 Key 在同一时间集体失效/过期`，或者 Redis 服务直接宕机。

此时，海量的请求全部涌向数据库，就像雪崩一样，瞬间将数据库压垮。

解决思路：**分散过期时间，避免集中失效。**

#### 方案一：随机化过期时间（最简单有效）
- 原理：在设置缓存的 TTL（过期时间）时，不要给所有 Key 设置相同的值。而是在基础过期时间上，加上一个随机值。
- 效果：例如基础过期时间是 1 小时，可以设置为 1小时 + 随机(1~5分钟)。这样大量 Key 的过期时间就会被错开，避免同一时刻集体失效。
#### 方案二：多级缓存
- 原理：在应用服务器本地（如 Caffeine、Guava Cache）和 Redis 之间建立多级缓存。
- 效果：即使 Redis 出现短暂故障或大面积失效，本地缓存依然能扛住一部分请求，起到缓冲作用。
#### 方案三：高可用架构（防宕机）
- 原理：如果雪崩是因为 Redis 服务器宕机引起的，那么需要从架构层面解决。
- 效果：使用 Redis 哨兵模式（Sentinel）或 Redis 集群（Cluster），确保 Redis 服务的高可用性，避免单点故障。
#### 方案四：服务降级与限流
- 原理：作为最后的兜底方案。当数据库压力过大时，触发限流（拒绝部分请求）或降级（直接返回默认值或错误提示），保护数据库不被彻底打死。


### 多级缓存

多级缓存（Multi-level Cache）是一种通过分层设计来提升系统性能的架构模式。

它的核心思想是按照“离用户/应用越近，速度越快，但容量越小、成本越高”的原则，将数据冗余存储在不同的存储介质中，从而在性能、成本和容量之间找到最佳平衡。

在典型的互联网后端应用中，多级缓存通常由以下三层构成：

#### 一级缓存 (L1) - 本地缓存
- 位置：与应用服务在同一个 JVM 进程内，例如使用 Caffeine、Guava Cache。
- 特点：访问速度极快（微秒级），但容量受限于应用服务器的内存大小，且无法在多个应用实例间共享。
- 用途：适合存储极热点的、不常变动的数据，如首页配置、字典表等。 

#### 二级缓存 (L2) - 分布式缓存
- 位置：独立部署的缓存服务集群，例如 Redis、Memcached。
- 特点：访问速度较快（毫秒级），容量远大于本地缓存，并且可以被所有应用实例共享。
- 用途：适合存储需要跨服务共享的热点数据，如用户 Session、商品详情等。
#### 三级缓存 (L3) - 数据库
- 位置：如 MySQL、PostgreSQL。
- 特点：访问速度最慢（百毫秒级），但数据持久化，是数据的最终来源和强一致性的保证。


:::tip 工作流程
1. 应用首先查询 L1 本地缓存。如果命中，则直接返回数据，流程结束。
2. 如果 L1 未命中，则查询 L2 分布式缓存。如果命中，则返回数据，并同步将数据回写到 L1，以便下次请求能更快获取。
3. 如果 L2 也未命中，则最终查询 L3 数据库。查到数据后，将数据回写到 L2 和 L1 两级缓存中，然后再返回给应用。
:::

## 事务

Redis 的事务机制和传统关系型数据库（如 MySQL）的事务有很大的不同。它的核心本质是一组命令的集合，通过特定的命令将多个操作打包，然后一次性、按顺序、排他地执行。

### 核心命令

- **MULTI**：开启事务。执行此命令后，客户端进入事务状态，后续输入的命令不会被立即执行，而是被放入一个队列中。
- **EXEC**：提交并执行事务。它会触发执行队列中所有已入队的命令。
- **DISCARD**：取消事务。它会清空事务队列，放弃执行所有已入队的命令。
- **WATCH**：监视一个或多个 Key。如果在 EXEC 执行前，被监视的 Key 被其他客户端修改，则当前事务会被中断，EXEC 不会执行。这是一种乐观锁的实现。

简单事务示例：

```bash
127.0.0.1:6379> MULTI
OK
127.0.0.1:6379> SET user:1001:name "Alice"
QUEUED
127.0.0.1:6379> INCR user:1001:login_count
QUEUED
127.0.0.1:6379> EXEC
1) OK
2) (integer) 1
```

**Redis 事务不支持回滚**，对于需要强原子性的复杂业务逻辑，通常推荐使用 `Lua 脚本` 来替代：

```lua
-- 假设用户下单需要同时完成两个操作：
-- 扣减账户余额（比如扣 100 元）
-- 增加商品销量（比如销量 +1）


-- 脚本逻辑：
-- KEYS[1] 是用户的余额 Key，ARGV[1] 是扣减金额
-- KEYS[2] 是商品的销量 Key，ARGV[2] 是增加的销量值

-- 1. 获取当前余额
local current_balance = tonumber(redis.call('GET', KEYS[1]))
local deduct_amount = tonumber(ARGV[1])

-- 2. 校验余额是否充足
if current_balance == nil or current_balance < deduct_amount then
    return -1 -- 返回 -1 表示余额不足，业务逻辑中直接终止
end

-- 3. 余额充足，执行扣减
redis.call('DECRBY', KEYS[1], deduct_amount)

-- 4. 增加商品销量
redis.call('INCRBY', KEYS[2], ARGV[2])

return 1 -- 返回 1 表示扣减成功
```

Redis中使用 EVAL 命令执行 Lua 脚本：

```bash
127.0.0.1:6379> EVAL "local current_balance = tonumber(redis.call('GET', KEYS[1]))
local deduct_amount = tonumber(ARGV[1])
if current_balance == nil or current_balance < deduct_amount then
    return -1
end
redis.call('DECRBY', KEYS[1], deduct_amount)
redis.call('INCRBY', KEYS[2], ARGV[2])
return 1" 2 user:1001:balance product:8848:sales 100 1
```
